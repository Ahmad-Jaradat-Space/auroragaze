# AuroraGaze — project rules

This file is the project's source of truth for an AI assistant working on
the codebase. Copy it verbatim to `CLAUDE.md` at the root of the new repo
when scaffolding. Read it before any code change.

Companion docs: `brief.md` (what + why), `architecture.md` (the design),
`build-plan.md` (week-by-week steps).

---

## What we're building

A solar-wind monitor that translates live data at L1 (DSCOVR / ACE / SOHO)
into two kinds of briefing:

1. Aurora-visibility briefings for chasers in southern Australia.
2. Storm-impact briefings for satellite operators (the Feb 2022 Starlink
   loss is the concrete anchor).

One repo. One live demo at `auroragaze.dev`. Three tagged releases
(v0.1 / v0.2 / v0.3). Top-pinned on the author's GitHub.

The repo is the artefact a hiring manager will read. Every commit is
written with that reader in mind.

---

## Hard rules

### Style

- The repo must read as written by a scientist, not generated by a model.
- Pydantic models for every I/O boundary. The schema is the doc.
- Functions over classes when a function will do.
- Type hints on every public function, return types included.
- One concept per file. 50–200 lines per module.
- No metaclasses, no clever decorators we wrote, no hidden control flow.
- Tests read like protocols: arrange, act, assert. Plain `assert`.
- `ruff` and `mypy --strict` gate every commit.

### Comments

- Default: no comment. Type hints + good names cover most cases.
- Comment only on a hidden constraint, a workaround, a non-obvious physics
  step, or a unit assumption (`# bz in nT, southward = negative`).
- Never restate what the code does.
- Never write "Step 1 / Step 2" or banner comments.
- Module docstrings are one line. Most functions don't get a docstring.
- No `TODO:` placeholders.

### Anti-AI-tells (the most important section)

Banned anywhere — code, comments, commits, READMEs, docs:

- **Banned words**: *robust, comprehensive, powerful, seamless, intuitive,
  cutting-edge, state-of-the-art, leverage, delve, elegant, sophisticated.*
- No emoji in code, commit messages, or commit bodies.
- No "Note:" / "Important:" headers in markdown.
- No co-author trailers in commits. No `Co-Authored-By: Claude`. No
  `Generated with Claude Code` footers. No AI signatures of any kind.
- No PR description templates unless the change actually warrants sections.
- README copy is present-tense and declarative. "Streams briefings over
  SSE" not "Provides streaming capabilities."

Variable names match the domain: `bz`, `kp`, `dst`, `v_kms`, `n_cm3`,
`f107`. Not `magnetic_field_z_component_in_nano_teslas`.

Before pushing a diff, read it out loud. If it sounds like marketing,
rewrite it. If a comment restates the code, delete it.

## Models and providers

| Role | Choice |
|---|---|
| Default LLM | DeepSeek V3 (`deepseek-chat`) via `ChatDeepSeek` |
| Alternative | MiniMax M2 via `ChatOpenAI` (OpenAI-compatible base URL) |
| Local / offline | Ollama (Llama 3.3 70B / Qwen 2.5 32B) via `ChatOllama` |
| Embeddings | self-hosted `BAAI/bge-large-en-v1.5` |
| Reranker | self-hosted `BAAI/bge-reranker-v2-m3` |
| Judge (eval) | DeepSeek V3 with strict rubric |

**Never default to Anthropic.** A single LLM factory in
`src/auroragaze/llm.py` reads `LLM_PROVIDER` from env so all three swap
with one variable.

Cost ceiling: $20/month total (LLM + Hetzner). If exceeded, push more
work to Ollama.

## How work flows

- Solo dev still uses feature branches and PRs (forces self-review).
- `main` is protected; no direct pushes.
- Conventional commits, lowercase: `feat:`, `fix:`, `docs:`, `refactor:`,
  `chore:`. One line. Body only when truly needed.
- Every PR runs CI: ruff, mypy, pytest, plus the eval / red-team / chaos
  gates that apply at that phase.
- One ADR per non-obvious choice. ADRs are one page: title, context,
  decision, consequences. Live in `docs/decisions/`.

## Working stages (the high-level plan)

Five stages from empty repo to a top-pinned demo on GitHub.

### Stage 0 — Scaffold (1 week)

Empty repo to a green CI run on a hello-world DeepSeek call.

- Directory tree, `pyproject.toml`, `docker-compose.yml`, `.env.example`.
- Pre-commit hooks (ruff, mypy).
- `src/auroragaze/llm.py` factory. CLI prints "hello" via DeepSeek.
- First tag: `v0.0` (optional internal milestone).

### Stage 1 — v0.1 — RAG done right (6 weeks)

Production-grade hybrid RAG inside a minimal LangGraph (supervisor →
retrieval → composer). No live data yet, no physics, no multi-agent.
The point is to nail RAG and the eval/CI loop.

Ship: a CLI that takes a query and returns a cited briefing from the
historical archive.

Success bar: recall@5 ≥ 0.85, reranked precision@3 ≥ 0.80, judge
κ ≥ 0.70, 100% briefings cite ≥ 1 chunk, zero hallucinated index values
on golden-set items, CI gate live on `main`.

Deliverables: `eval/golden_set.jsonl` (60 curated events),
`run_eval.py`, calibrated `judge.py`, `regression.yml` workflow,
v0.1 blog post draft, tag `v0.1`.

### Stage 2 — v0.2 — Multi-agent + tools + MCP (6 weeks)

Full 5-specialist LangGraph with live data, both personas, streamed
agent trace in the browser, MCP server, red-team coverage, chaos test.

Ship: a hosted `/aurora` and `/satellite` page that streams the agent's
tool calls and the final briefing live.

Success bar: 5 specialists invoked across persona paths (≥2 in parallel
where dependencies allow), 8 tools all Pydantic-typed, 30/30 red-team
traces with zero fabricated numbers, MCP server installable in Claude
Desktop in one command, chaos test green (DSCOVR-down → ACE fallback
surfaced in the briefing).

Deliverables: `feeds/ingester.py`, `tools/*.py`, all five specialists,
`mcp_server/`, `redteam/`, `reliability/chaos_test.py`, frontend hero
+ briefing pages, v0.2 blog post draft, tag `v0.2`.

### Stage 3 — v0.3 — Production deploy + cost discipline (4 weeks)

Hosted live demo with prompt caching, full observability, real cost
economics, blue/green CI deploy, one chaos drill in production.

Ship: `auroragaze.dev` live, public Grafana metrics board, `economics.md`
with real numbers, blue/green workflow.

Success bar: live URL up and monitored, ≥ 90% prompt-cache hit rate on
per-region aurora playbooks, p50 < 4s and p95 < 10s for a streamed
briefing, cost per briefing ≤ $0.02 cached / ≤ $0.05 cold, total
monthly run ≤ $20.

Deliverables: `deploy/compose.prod.yml`, `deploy/hetzner.md`, Langfuse
self-hosted, `economics.md`, `deploy.yml` workflow, v0.3 blog post
draft, tag `v0.3`.

### Stage 4 — Launch on GitHub (1 week)

Polish the public-facing surface so a hiring manager forms a strong
opinion in 4 minutes.

- README at production quality: 30-second Loom, live demo URL, three-
  release table, screenshot of streamed-trace UI, eval numbers, cost
  numbers, "why I built this" footer (geodesy + space-weather
  background, Feb 2022 Starlink event, southern aurora community).
- Three blog posts published, linked from README and from each release.
- Pin the repo on the GitHub profile.
- One ADR per release in `docs/decisions/` explaining a non-obvious
  choice (e.g. "why DeepSeek over Anthropic," "why hybrid retriever
  not just dense," "why pgvector not Pinecone").
- A handful of small issues opened (real follow-ups, not theatre) — a
  living repo reads more credibly than a finished one.

Total elapsed: 18 weeks at the planned cadence. Faster is possible by
compressing weekends into the schedule, but not at the cost of any
success bar.

## Definition of done (every stage)

- All tests green.
- All CI gates green (ruff, mypy, pytest, plus the gates for that
  stage).
- Tag pushed to `main`.
- Changelog updated.
- README reflects the new state.
- ADR written for at least one non-obvious choice introduced this stage.
- Blog post draft committed (publish at Stage 4).

## What this assistant must never do

- Default to Anthropic models or use `langchain-anthropic`.
- Add co-author trailers, AI signatures, or generated-by footers to
  commits or PRs.
- Use any banned word from the anti-AI-tells list.
- Add banner comments, `TODO:` placeholders, or "Note:" / "Important:"
  headers.
- Restate code in comments.
- Push directly to `main`.
- Skip the eval / red-team / chaos gates that apply at the current stage.
- Add features beyond the brief without an ADR justifying them.
- Use "comprehensive", "robust", "powerful", "seamless", "elegant",
  "leverage", "delve", or any other marketing verb anywhere.

---

## What this assistant should always do

- Read `brief.md`, `architecture.md`, and the relevant section of
  `build-plan.md` before changing code in a new area.
- Prefer editing existing files over creating new ones.
- Keep modules under 200 lines; split when they grow.
- Validate every external boundary with Pydantic.
- When unsure between two designs, write the simpler one and an ADR.
- Treat the README as code. Edit with the same care as `src/`.
